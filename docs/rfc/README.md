# RFCs — Request For Comments

Longer proposals for non-trivial changes that need thinking-out-loud before code. Most decisions in this project skip the RFC stage and go straight to ADR — reach for RFC only when the answer isn't obvious and you'd want to sit on it for a day.

## Lifecycle

1. **Draft** — you're still writing it. Status: `draft`.
2. **Open for comment** — ready for input (from your future self, advisors, collaborators). Status: `open`.
3. **Resolved** — decision made. Status: `accepted` (write the ADR) or `rejected` (keep the RFC for the trail).

## Rules

- Filenames: `NNNN-kebab-case-title.md`, zero-padded 4 digits, separate counter from ADRs.
- Once accepted → spawn an ADR with the *decision*, leave the RFC as the *reasoning trail*. The ADR cross-links the RFC.
- If rejected, leave the RFC in place with `status: rejected` and a one-line note — future-you will thank you.
